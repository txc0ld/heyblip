## Heavy Runtime Surfaces

### 1. Startup fan-out is broad and front-loads many services

Primary file: [AppCoordinator.swift](../../Sources/Services/AppCoordinator.swift#L438)

Cold start currently kicks off all of the following from `start()`:

- transport start
- location authorization and geohash updates
- message retry service start
- notification authorization
- profile load
- channel load
- mesh monitoring start
- location monitoring start
- event load
- geofencing start
- StoreKit startup
- SOS responder status load
- visible SOS refresh
- one-shot presence broadcast after 3 seconds
- repeating presence broadcast every 30 seconds
- stale-peer prune every 60 seconds
- message cleanup service startup

This is the highest-confidence pre-sprint runtime hotspot because it compounds startup CPU, wakeups, networking, BLE, SwiftData fetches, and log output in one path.

### 2. Nearby idle has multiple overlapping timers and refresh loops

Primary files:

- [NearbyView.swift](../../Sources/Views/Tabs/NearbyTab/NearbyView.swift#L468)
- [MeshViewModel.swift](../../Sources/ViewModels/MeshViewModel.swift#L181)
- [LocationViewModel.swift](../../Sources/ViewModels/LocationViewModel.swift#L158)
- [FriendFinderViewModel.swift](../../Sources/ViewModels/FriendFinderViewModel.swift#L166)
- [FriendFinderMapView.swift](../../Sources/Views/Tabs/NearbyTab/FriendFinderMapView.swift#L450)

Notable idle work while Nearby and Friend Finder are open:

- `NearbyView` starts presence broadcasts immediately, then every 10 seconds when visibility is enabled.
- `MeshViewModel` refreshes mesh state every 5 seconds.
- `LocationViewModel` refreshes friend locations and user location every 10 seconds.
- `FriendFinderViewModel` runs stale peer cleanup every 10 seconds.
- `FriendFinderMapView` refreshes the location snapshot every 10 seconds and can also broadcast location when sharing is enabled.

This is the clearest candidate for elevated idle CPU and energy while the user sits in Nearby with visibility enabled.

### 3. Chat and voice-note flows are timer-heavy and queue-heavy

Primary files:

- [MessageService.swift](../../Sources/Services/MessageService.swift#L40)
- [ChatViewModel.swift](../../Sources/ViewModels/ChatViewModel.swift#L341)
- [ChatView.swift](../../Sources/Views/Tabs/ChatsTab/ChatView.swift#L329)
- [AudioService.swift](../../Sources/Services/AudioService.swift#L32)

Before cleanup, the main chat path includes:

- a user-initiated message processing queue
- a separate user-initiated encryption queue
- Noise handshake setup and queued-message flushes
- delivery and read receipts
- typing indicator timers
- voice-note recording/playback timers at 50 ms cadence
- PTT chunk emission timer at 20 ms cadence

Even without a trace, this is the most likely hot path for short CPU bursts and radio/audio energy spikes during message send and voice-note send/receive.

### 4. Cleanup and sync services add periodic background wakeups

Primary files:

- [MessageCleanupService.swift](../../Sources/Services/MessageCleanupService.swift#L29)
- [StateSyncService.swift](../../Sources/Services/StateSyncService.swift#L38)

Observed cadence:

- message cleanup sweep starts immediately and then repeats every 5 minutes
- state sync repeats every 15 minutes, or 30 minutes under low-power mode

These are not likely to dominate active interaction cost, but they matter for idle wakeups and energy when stacked on top of mesh/location work.

## Real-Device Profiling Steps To Run

These are the exact steps to use once a physical iPhone is connected.

### Time Profiler

1. Connect the iPhone and confirm it is online in `xcrun xctrace list devices`.
2. Open the project in Xcode.
3. Select the connected iPhone as the run destination.
4. Use `Product > Profile` so Instruments launches a profiled build.
5. Choose `Time Profiler`.
6. Run each flow separately, not mixed together:
   - cold launch to main tab shell, then idle 3 minutes
   - Nearby with visibility enabled, then Friend Finder idle 3 minutes
   - 5 to 10 text messages plus 1 voice note send/receive
   - final idle window for 3 minutes
7. For each capture, keep a note of:
   - top self-weight stack
   - top cumulative stack
   - dominant subsystem or file
   - whether CPU stays elevated during idle

### Energy Log

1. Repeat the same four flows with the `Energy Log` template.
2. Note:
   - CPU energy impact
   - networking energy impact
   - location energy impact
   - audio energy impact
   - any sustained wakeups during idle

### Heat And Battery

Use the same physical device for both pre-fix and post-fix reruns.

Record:

- battery percentage before flow 1
- battery percentage after flow 4
- whether the device feels cool, warm, hot, or thermally constrained
- any thermal warnings in Xcode or on-device

## Current Conclusion

The baseline is only partially captured from this session because no online iPhone was available. The strongest pre-fix risk areas are already clear from the code:

- startup work is too fan-out heavy
- Nearby idle layers several periodic refresh loops
- messaging and audio flows combine queue churn, encryption, radio work, and frequent timers
- debug logging is densest in the same message and startup paths likely to be hot

The next step for this ticket is not a code change. It is a manual real-device Instruments pass using the steps above, then filling in the missing CPU, energy, heat, and battery deltas with trace-backed numbers.
